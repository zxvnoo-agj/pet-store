## User Scenarios & Testing *(mandatory)*

### User Story 1 - Import Crawled Product Data from TXT Files (Priority: P1)

运营人员通过第三方爬虫工具生成商品数据txt文件（存放于`pet-store/pdd`目录），然后通过管理后台将批量txt文件内容提取导入到数据库中，形成可供后续匹配使用的爬取数据库。

**Why this priority**: 爬取数据入库是整个优化流程的前提。没有爬取数据，后续的goods_id匹配和LLM提取都无法进行。

**Independent Test**: 运营人员将一批txt文件放入指定目录后，触发导入操作，能在管理后台看到导入结果（导入成功数、跳过数、失败数），并在数据库中确认数据已入库。

**Acceptance Scenarios**:

1. **Given** `pet-store/pdd`目录下存在N个爬虫生成的txt文件，**When** 运营人员在管理后台触发"导入爬取数据"，**Then** 系统扫描目录下所有txt文件，逐个解析内容并写入数据库爬取数据表
2. **Given** 某个txt文件格式不符合预期或内容解析失败，**When** 系统处理该文件，**Then** 系统记录该文件的失败原因（如"格式错误"、"goods_id缺失"），并继续处理后续文件，不影响整体导入流程
3. **Given** 某个goods_id已在爬取数据库中存在，**When** 系统导入该商品数据，**Then** 系统更新已有记录（覆盖旧数据），并在导入结果中标记为"已更新"
4. **Given** 导入操作完成，**When** 运营人员查看导入结果，**Then** 系统展示本次导入汇总：新入库N条、更新M条、失败K条（含失败文件列表和原因）

### User Story 2 - Match Strategy Search Results with Crawled Data and Enrich via LLM (Priority: P1)

策略搜索任务完成后，系统对搜索结果中的每个商品，通过goods_id匹配爬取数据库。对于匹配到的商品，系统调用视觉/自然语言大模型从爬取内容中提取结构化字段，再通过goods.detail补充剩余信息，完成单个商品的完整采集。

**Why this priority**: 这是商品数据从"发现"到"可用"的核心转换环节。没有LLM提取和详情补全，采集到的商品数据将缺乏可展示的结构化信息。

**Independent Test**: 对一个已存在于爬取数据库中的goods_id，手动触发匹配和提取流程后，能在5分钟内看到该商品的状态变为"已采集（active）"，且所有结构化字段（品牌、规格、价格等）已填充。

**Acceptance Scenarios**:

1. **Given** 策略搜索任务已返回商品列表（含goods_id），**When** 系统对每个商品执行匹配流程，**Then** 系统在爬取数据库中按goods_id查找，匹配到则进入LLM提取流程，未匹配到则标记为"待采集"
2. **Given** 某个商品在爬取数据库中匹配成功，**When** 系统触发LLM提取，**Then** 系统将爬取的原始内容（文本/图片）提交给大模型，按预设字段要求提取结构化信息（品牌、规格、形态、产地、保质期、适用年龄、成分、营养亮点等）
3. **Given** LLM提取完成，**When** 提取结果返回，**Then** 系统将提取的结构化字段写入商品记录，并记录提取置信度
4. **Given** LLM提取后仍存在缺失字段，**When** 系统调用goods.detail接口，**Then** 系统获取商品详情并补全剩余缺失字段（如当前价格、销量、店铺信息等）
5. **Given** LLM提取和goods.detail补全均已完成，**When** 单个商品处理结束，**Then** 系统将商品状态更新为"active"（已采集），表示该商品数据完整可用

### User Story 3 - Mark Unmatched Products as Pending Collection (Priority: P2)

对于策略搜索发现但在爬取数据库中不存在的商品，系统将其标记为"待采集"状态，运营人员可在管理后台查看待采集商品列表，后续通过其他方式补充采集。

**Why this priority**: 状态标记确保数据完整性可追踪。运营人员可以清楚知道哪些商品尚未采集数据，便于后续安排手动或批量补采。

**Independent Test**: 运营人员在管理后台查看待采集商品列表，能看到所有状态为"pending"的商品及对应的goods_id和发现时间。

**Acceptance Scenarios**:

1. **Given** 策略搜索返回的某商品在爬取数据库中无匹配，**When** 系统完成匹配判断，**Then** 系统将商品状态设为"pending"（待采集），并在商品记录中注明"爬取数据未覆盖"
2. **Given** 管理后台存在一批待采集商品，**When** 运营人员查看待采集列表，**Then** 系统展示商品ID、名称、goods_id、发现时间、来源策略等信息
3. **Given** 某个待采集商品后续被爬虫覆盖（新一轮txt导入后goods_id出现在爬取数据库中），**When** 运营人员对该商品手动触发"重新匹配"，**Then** 系统重新执行匹配→LLM提取→详情补全流程，将商品状态更新为"active"

### User Story 4 - View Enrichment Results and LLM Extraction Quality (Priority: P3)

运营人员在管理后台能够查看每次匹配提取任务的执行结果，包括成功提取的商品数、LLM提取的各字段完成率、以及提取失败的记录详情。

**Why this priority**: 运营监控是辅助功能，帮助运营人员了解数据采集质量和LLM提取效果，但即使没有此页面，核心采集流程仍可正常运行。

**Independent Test**: 一次批量匹配提取任务完成后，运营人员能在"提取日志"页面看到本次任务汇总及各商品详情。

**Acceptance Scenarios**:

1. **Given** 一次匹配提取任务已执行完成，**When** 运营人员打开提取日志页面，**Then** 系统展示任务概览（总处理数、成功数、失败数、LLM字段完成率）和商品级别处理详情
2. **Given** 某个商品LLM提取部分字段失败，**When** 运营人员查看该商品提取详情，**Then** 系统展示哪些字段提取成功、哪些字段缺失及其原因

### Edge Cases

- **txt文件编码异常**: 爬虫生成的txt文件可能包含非UTF-8编码内容。系统应自动检测编码（支持GBK/UTF-8），编码检测失败时标记文件为导入失败并记录原因
- **txt文件内容截断或不完整**: 爬虫输出可能因网络中断等原因导致txt文件内容不完整。系统解析时检测必填字段（如goods_id）是否缺失，缺失则标记为失败
- **LLM返回格式异常**: LLM返回非预期格式或缺少关键字段时，系统使用已有字段写入，缺失字段留空，不中断整个提取流程
- **goods.detail接口超时或失败**: 调用goods.detail补全时如遇超时或错误，系统记录失败信息，已提取的LLM字段保留，商品标记为部分完成状态，支持重试
- **同一goods_id重复出现**: 爬取数据库中goods_id唯一；导入时后导入的数据覆盖先导入的数据。策略搜索中同一goods_id多次出现时，系统自动去重，仅处理一次
- **批量导入文件数量极大**: 当目录下存在数百个txt文件时，系统采用分页批量处理，避免内存溢出，并展示实时进度
- **LLM服务不可用**: LLM服务不可用时，系统记录错误并跳过当前商品，标记为"提取失败"，待LLM恢复后运营人员可手动重试
- **爬取内容包含非商品信息**: 爬虫可能意外爬取到非商品页面内容（如广告、活动页）。系统在LLM提取前通过内容长度和关键词初步过滤，明显非商品内容标记为"待审核"

## Requirements *(mandatory)*

### Functional Requirements

**Explicitly Out of Scope**: Modification of existing product attribute fields in the database. Direct Playwright-based scraping (replaced by third-party tool). Automatic scheduling of the third-party crawler tool (executed manually by operators). Changes to the existing product browsing, AI assistant, or mini program frontend.

- **FR-001**: System MUST support importing product data from txt files generated by a third-party crawler tool, located in the `pet-store/pdd` directory. Operators trigger import from the admin backend; the import runs synchronously and returns results immediately upon completion.
- **FR-002**: System MUST parse txt file content and extract structured fields into the crawled product database, using goods_id as the unique key for deduplication (re-import updates existing records).
- **FR-003**: System MUST report import results including: total files processed, new records created, records updated, files failed, and per-file failure reasons.
- **FR-004**: After a strategy search task completes, system MUST attempt to match each discovered product's goods_id against the crawled product database.
- **FR-005**: For matched products, system MUST invoke a visual/natural-language LLM to extract the following structured product fields from crawled content (text and images): brand, spec_weight, spec_form, origin, shelf_life, age_range, special_formula, top-8 ingredients, nutrition_highlight.
- **FR-006**: After LLM extraction, system MUST invoke goods.detail to supplement any remaining missing product fields (e.g., current price, sales volume, store information).
- **FR-007**: Product collection status MUST progress through the following states: "pending" (awaiting matching/collection) → "enriching" (LLM extraction or detail fetch in progress) → "active" (fully collected and ready). Failures during enrichment transition to "failed" status with manual retry capability.
- **FR-008**: For products not found in the crawled product database after strategy search, system MUST mark them as "pending" with a note indicating that crawled data does not cover this product.
- **FR-009**: System MUST NOT modify existing product attribute field definitions in the database (table columns remain unchanged per Feature 001/002 data models).
- **FR-010**: System MUST handle txt file encoding detection (UTF-8 and GBK) and report encoding issues as import failures.
- **FR-011**: System MUST log all enrichment task executions with: goods_id, match status (matched/unmatched), LLM extraction result (success/partial/failed), detail fetch result, and final product status.
- **FR-012**: Operators MUST be able to view products in "pending" status and manually trigger re-matching for individual products from the admin backend.
- **FR-013**: Operators MUST be able to view enrichment task logs showing per-task and per-product enrichment results from the admin backend.

### Key Entities *(include if feature involves data)*

- **Crawled Product Record** (new dedicated table `crawled_products`): Stores raw and parsed product data imported from third-party crawler txt files. Key fields: goods_id (unique key), raw content (original txt content), parsed fields extracted during import, crawl timestamp, file source, and import timestamp. Separate from `external_products` to isolate raw crawl data from operational platform mappings.
- **Enrichment Log**: Records the execution of product enrichment tasks (LLM extraction + goods.detail completion). Tracks goods_id, match status, LLM extraction status, fields extracted, fields missing, detail fetch status, and final product status.
- **Product (Existing)**: The existing Product entity from Feature 001/002 data model. Product attribute columns remain unchanged. Collection status field is extended with additional transitional states as needed.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator can import a batch of 50 txt files into the crawled product database within 5 minutes of triggering the import operation
- **SC-002**: 95% of valid txt files (correctly formatted, with goods_id) are successfully parsed and imported on first attempt
- **SC-003**: After strategy search completes, product matching against the crawled database completes within 1 second per 100 records
- **SC-004**: LLM extraction produces complete structured fields (≥80% of predefined fields filled) for 85% of products with adequate crawled content
- **SC-005**: A single product goes from "pending" to "active" (full collection complete) within 3 minutes of triggering the enrichment flow
- **SC-006**: Products without crawled data coverage are correctly marked as "pending" — zero false-positive matches (goods_id mismatch)
- **SC-007**: Operators can view product enrichment results and identify extraction quality issues within 2 clicks from the admin dashboard
- **SC-008**: Existing product attribute fields remain unchanged after the refinement — schema validation confirms 100% backward compatibility

- Q: How should 003's pipeline relate to 002's existing enrichment (PDD detail API → LLM from PDD title)? → A: 003 replaces 002's enrichment for matched products; 002's pipeline continues to serve unmatched/pending products or as fallback
- Q: What are ALL the fields the LLM should extract from crawled content (resolving "and other predefined fields")? → A: Complete set matching 002 FR-003: brand, spec_weight, spec_form, origin, shelf_life, age_range, special_formula, top-8 ingredients, nutrition_highlight
- Q: What happens to products already in the DB collected via the old Playwright scraper? → A: Existing products retain current data and status; only products discovered by strategy searches after 003 deployment use the new pipeline

## Assumptions

- The third-party crawler tool is executed manually by operators and its output (txt files) follows a known format that can be defined and documented for parsing
- The crawler tool outputs one txt file per product, or a batch file with a defined delimiter, placed in the `pet-store/pdd` directory
- The existing `search_strategies` table and strategy search workflow from Feature 002 are reused as-is (this feature refines the data ingestion path, not the search mechanism)
- For products matched in the crawled database, 003's enrichment pipeline (crawled content → LLM → goods.detail) replaces 002's enrichment pipeline (PDD detail API → LLM from title). 002's pipeline remains available for unmatched/pending products as a fallback enrichment path.
- Existing products in the database (previously collected via Playwright or other means) retain their current data and status. Only products discovered by strategy searches after 003 deployment are processed through the new crawled-database pipeline.
- LLM service (visual and natural language) is available and configured with sufficient context window to process crawled product content
- The `goods.detail` interface (already defined or reusing Feature 002's PDD detail fetch or equivalent) remains available for supplementary data
- Product attribute table columns defined in Feature 001/002 data models are preserved as-is; new fields, if any, are added via existing JSONB columns (`specifications`, `ratings`) or new columns that do not modify or remove existing ones
- The existing product status lifecycle (pending → enriching → active → failed) from Feature 002 is reused and extended where needed
- Admin backend infrastructure from Feature 001/002 is reused for the new import and enrichment monitoring pages
