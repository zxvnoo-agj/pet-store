# Delta Implementation Plan: Data Collection Module v3

**Branch**: `002-data-collection-module` | **Date**: 2026-05-15
**Base Plan**: [plan.md](plan.md) (v2, fully implemented)
**Delta Spec**: [delta-spec.md](delta-spec.md)

## Summary

Incremental changes to the already-implemented 002 data collection module: introduce httpx-based PDD page scraping with session cookies for rich detail extraction (images, SKU, mall info), add vision LLM (Qwen-VL-Plus) for ingredient/nutrition recognition from product images, implement PDD promotion URL generation with caching for commission tracking, and expand the hourly price refresh to include `promotion_rate`. The existing search strategy + PDD search API discovery layer is preserved unchanged.

## Technical Context

**Language/Version**: Python 3.11+ (backend, same as v2), TypeScript 5.x (admin frontend, same as v2)
**Primary Dependencies**: FastAPI 0.110+ (existing), SQLAlchemy 2.0 async (existing), **httpx** (NEW, for PDD page scraping), **openai** (NEW, for Qwen-VL), aiohttp (existing), APScheduler (existing), **playwright** (NEW, login script only)
**Storage**: PostgreSQL 15 (existing — new columns + 1 new table), Redis (optional, for promotion URL cache acceleration)
**Testing**: pytest + pytest-asyncio (backend), jest (admin frontend)
**Target Platform**: Linux server (backend), WeChat Mini Program + H5 Admin
**Project Type**: web-service extension (backend) + admin UI extension
**Performance Goals**: Single httpx request < 10s per product (incl. 8-15s delay), vision LLM batch analysis of ~5 images per product < 2min, promotion URL < 2s cached / < 5s API
**Constraints**: 8-15s inter-request delay (conservative); PDD API 2000 calls/day shared across search + price refresh + promotion URLs; Qwen-VL-Plus costs ~¥0.0015/1k tokens
**Scale/Scope**: MVP ~500 products, 2 new services (crawler + vision), 1 new public endpoint, ~9 new DB columns + 1 new table

## Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|------------|
| PDD changes `window.rawData` structure | Medium | High — crawler returns empty data | Design crawler to log raw data structure on failure; quick patch possible due to single extraction point |
| PDD session cookies expire | Medium | Low — cookie refresh via login script | `login_pdd.py` re-exports cookies; `save_session.py` as CDP-based fallback |
| PDD anti-bot detection via TLS fingerprint | Low | Low — `httpx` has no browser fingerprints to detect | If detected, fallback to Playwright CDP approach (previous architecture) |
| Qwen-VL API quota exhausted | Low | Low — ingredients/nutrition fields empty | Pipeline degrades gracefully; vision step is non-blocking for ACTIVE transition |
| Promotion URL cache miss cascade on Redis failure | Low | Medium — PDD API rate limit hit | PostgreSQL fallback cache table; Redis is optional acceleration layer |
| PDD API 2000/day quota pressure from 3 consumers | Medium | Medium — some requests throttled | Priority: price refresh > promotion URL > search discovery; monitor via Prometheus |

## Complexity Tracking

No constitution violations. All extensions align with existing patterns (async services, Pydantic schemas, SQLAlchemy models, Alembic migrations).

| Aspect | Decision | Justification |
|--------|----------|---------------|
| Cookie-based HTTP scraping | httpx.AsyncClient with session cookies | No browser process, no fingerprint detection, faster and simpler than Playwright |
| Promotion URL cache: PG primary | PostgreSQL table with UNIQUE(goods_id, pid) | Avoids hard Redis dependency; Redis added as optional acceleration layer |
| Pipeline degradation | Steps fail independently, product still → ACTIVE | Ensures product availability even if partial enrichment fails |
| Two-tier storage | Top-level columns + JSONB | Queryable fields indexed; bulk data flexible |
